Abstract

This paper takes a greenfield approach to envisioning an ideal Reusable Learning Object ecosystem. It suggests ease of development, ease of distribution and access, and relevance as requirements of such a system. It concludes with five actionable recommendations for stakeholders: encourage flexibility in learning object development processes, cite or deploy learning objects as web documents, embed learning object metadata within the learning object itself, improve the learning object search experience, and facilitate crowdsourced contributions to learning objects and their metadata.

Introduction

What are Reusable Learning Objects?

Reusable Learning Objects (RLO's) are "(digital media) representations designed to afford uses in different educational contexts".1 Examples include digitally delivered presentations, practice exercises, simulations, conceptual models, information, and contextual representations. They can take many forms and be implemented in many different ways. This definition is general enough to allow for a large amount of flexibility in RLO design.

Reusable Learning Objects can -- and should -- have metadata associated with them. This helps make information about the RLO -- like it's title, author, subject matter, and intended audience -- more accessible. Various RLO metadata format standards exist.2 Such metadata can be written manually using a text editor or metadata editor tool, or generated automatically from the learning object by an automatic metadata generator.

RLO's are often distributed through repositories. Like metadata formats, various learning object repositories also exist.3 These repositories range from public and general purpose to private and specialized. Most repositories will index any provided metadata and index it. Some will automatically generate metadata from learning objects.

Reusable Learning Objects, their metadata, and learning object repositories comprise a complete ecosystem of digital learning materials. Educational content developers can distribute RLO's, annotate them with metadata, and distribute them to relevant repositories. Teachers and learners can then search those repositories for learning objects supporting their educational objectives. In corporate environments, these tools can support the distribution of educational materials to a diverse workforce.

Discussion

Motivation for Best Practices

Born more of computer science than education, RLO's have a broad and abstract definition -- sometimes too abstract. This can make it difficult for RLO developers to move quickly from concept to delivery, and for users to quickly find useful learning objects.

XKCD: The General Problem

In an Ideal World, Learning Objects Would...

  • Be easy to develop
  • Be easy to distribute and access
  • Be relevant

As of 2016, how should we approach these goals?

Ease of Development

The broad definition of learning objects allows for an equally broad group of developers, tools, and workflows to develop them. As such, learning object developers should use whatever development processes they are most comfortable and productive with. A broad developer base using their favourite tools should produce the most diverse quantity of high quality learning objects possible.

Ease of Distribution and Access

Easy distribution of and access to learning objects combines two subgoals:

  • An easy distribution workflow for content creators
  • Easy content discovery, acquisition, and consumption by consumers

We consider consumers needs first.

Easy Access

Web browser users comprise an expansive audience. Search engines are a common port of call,4 and make diverse material accessible through an excellent interface. The learning object delivery model should replicate this structure as closely as possible.

Ideally, a user would open their preferred browser on their preferred device, navigate to a learning object search engine, and search for content. To support this discovery model, learning object repositories should strive to provide an accessible search interface. They should also make thel earning objects they host and their metadata accessible to web crawlers and users in order to support the development of third party learning object search engines.

A particularly useful model is that of Google Scholar and the academic literature. Google Scholar indexes metadata from html meta tags in citation pages for journal articles. Citation pages then provide access to the journal articles being indexed. This model would fit well for learning objects, as it allows for metadata to be embedded in the citation page, as well as for diverse media formats to be cited.

To summarize, easy content discovery, acquisition, and consumption by consumers is best supported by:

  1. Web-accessible learning objects annotated with HTML-embedded metadata within the learning object itself and/or its citation page

  2. Accessible and effective search engines for finding these objects

Easy Distribution

Consumer preferences for learning object accessibility informs what an effective distribution workflow for content creators should look like. In support of (1) above, learning object developers should strive to distribute their work as HTML documents, or as assets embedded in HTML documents. In support of (2), relevant learning object metadata should also be embedded within these documents. Beyond this, as Google Scholar and the academic literature demonstrate, significant flexibility exists in terms of how learning objects can be hosted.

Modern browsers are ubiquitous, fast, self-updating, and can deliver complex applications and rich media. Learning objects distributed as javascript, HTML, and CSS assets are likely to have maximum portability moving forward. Learning object developers should exploit this opportunity to reach the broadest possible audience.

Relevance

A relevant learning object should meet two requirements:

  1. Its metadata is comprehensive and accurate
  2. It's content is correct and up-to-date

Ensuring relevant metadata may require ongoing maintenance. To the extent that learning object discovery patterns evolve, metadata should be updated to service these new patterns. For example, if an existing technique begins to be called by a new name, metadata should be updated to reflect this new name.

Unfortunately, learning object developers and repository managers cannot be expected to perform this maintenance. Developers may become unavailable for various reasons, and repository managers may not have the expertise to accurately assess the maintenance needs for a particular learning object. Responsibility for metadata maintenance must fall elsewhere.

Two groups are well positioned to perform this maintenance: educational users and search engines. Crowdsourced information tends to be more accurate and reliable than expert assessment.5 Search engine technology can be developed to map queries to learning objects even with unmaintained metadata. Educators and learners can learn to use related queries to find related learning objects.

To support these requirements, learning object repositories should seek to engage users in updating and maintaining learning object metadata, and learning object content. Leadership in learning object search is also required.

Conclusion

Progress towards a robust learning object ecosystem requires that learning objects:

  • Be easy to develop
  • Be easy to distribute and access
  • Be relevant

Striving towards the following goals can help meet these requirements:

  • Minimize constraints on the learning object development process
  • Distribute learning objects either as native web applications or documents, or as assets acessible via a dedicated citation webpage
  • Embed learning object metadata within the learning object and its citation page
  • Strive for excellence in learning object search engines
  • Enable crowdsourced updates to learning objects and their metadata
